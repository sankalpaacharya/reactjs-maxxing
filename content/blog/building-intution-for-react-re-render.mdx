# Recursive Rendering

React 16 introduced the new reconciliation algorithm, old reconciliation algorithm used to render everything 
recursively, meaning once the rendering process starts there is no way for us to stop it unless it finishes the work
itself. Now that led to some issues.

<img src="/blog/scheduler/angry.svg" alt="React UI Runtime" width="400" height="100" style={{ display: 'block', margin: '2rem auto' }} />

say your browser is working at `60fps`, which is 60 frames per second, 1 frame in `16.6ms`. your browser 
has to render `1` frame in `16ms`. during this one rendering process it needs to reflow, repaint etc.
if your javascript itself takes more than `16ms` or even `16ms`, it is not giving time for the browser to do the work. it would drop the 
frame, which would lead to janky UI.

# New Reconciliation algorithm

<img src="/blog/scheduler/zoro.svg" alt="React UI Runtime" width="800" height="100" style={{ display: 'block', margin: '2rem auto' }} />

so with this new reconciliation algorithm react breaks 
the rendering process into a small unit of work, rather than doing all rendering process at once we perform a rendering 
in a small unit of work in a time slice

<img src="/blog/scheduler/wait.svg" alt="React UI Runtime" width="800" height="100" style={{ display: 'block', margin: '2rem auto' }} />

React is just like "_I will do the work I can in this time frame(~5ms) and will let the browser do his work_ " once browser finishes his work 
it will again let react know _I'm done with my work, you can continue now_. Now react will resume its rendering work from where it had left off. 


# Fibers
When you write JSX, you are describing how your UI should look. JSX is just syntactic sugar for `React.createElement`. A compiler like Babel transforms JSX into JavaScript function calls, which at runtime produce React elements. During the reconciliation process, React takes these React elements and incrementally creates [Fiber](https://github.com/facebook/react/blob/main/packages/react-reconciler/src/ReactFiber.js#L138-L211) nodes from them.

```tsx
const fiber = {
  type: "form",
  key: null,

  // tree structure
  child: null,
  sibling: null,
  return: null,

  // state & updates
  memoizedState: null,

  lanes: 0,
  childLanes: 0,

  // side effects
  flags: 0,

  // alternate for diffing (current <-> workInProgress)
  alternate: null
};
```

- See those three properties of fiber object? `child`, `sibling`, `return`. These are pointers. They connect fibers to other fibers, forming a linked list.


<img src="/blog/fiber/fiber.svg" alt="React UI Runtime" width="800" height="100" style={{ display: 'block', margin: '2rem auto' }} />

Fiber is nothing but just javascript object that is an in-memory representation of how your UI looks, 
During the initial mount of the components react will build this fiber tree off the react elements. react 
reconciliation algorithm only cares about the react fiber, it has no clue of the DOM at all. this platform agnostic 
data structure allows react to be used on other platforms as well like building ui for [mobile app](https://github.com/facebook/react-native), [3d app](https://github.com/pmndrs/react-three-fiber), [terminal app](https://github.com/vadimdemedes/ink)

# Four Phases: Trigger, Schedule, Render and Commit

The whole core point of the react fiber architecture is being able to do a big rendering process 
into a small unit of work, the main advantage of breaking rendering process into a small unit of work 
is that we are not blocking the browser from putting any task into a callstack.  

- [Understanding why react fiber exists](https://inside-react.vercel.app/blog/understanding-why-react-fiber-exists)

Let's see how things exactly work.

## Trigger

say react is rendering a product list by fetching an api and in middle you clicked an `increment` button your callback `()=>setCount(count+1)` is registered into a macro queue
in the browser. But since react works in a time slice it internally keeps checking if i can process more unit of work, it sees 
the time `5ms` is up and now i should let the browser yield. now browser will put that onClick callback 
function into a callstack. 


<img src="/blog/scheduler/done.svg" alt="React UI Runtime" width="800" height="100" style={{ display: 'block', margin: '2rem auto' }} />

your callback function has a `setCount` function inside of it, `setCount` is nothing but a state dispatcher internally.

## Schedule

now the task is to schedule a new render, you may be wondering why schedule a
new render? we haven't completed the one we were working on, right? True, we will answer in a bit

react now has to figure out the priority for this update we just triggered 
by calling `setCount` and priority in react is decided by the lane. Go above in this blog and take a look at the fiber properties. There are properties `lanes` and `childLanes`

- lanes: for the pending work of itself
- childLanes: for the pending work of its subtree

> Lanes in React are a 32-bit integer-based priority system introduced to manage concurrent rendering, acting as "virtual highways" that categorize the urgency of updates
<InfoNote>
React uses a 32-bit integer to represent lanes, which allows multiple lanes (priorities) to be merged together efficiently. For example:
```tsx 
const SyncLane = 1;
const TransitionLanes = 2;

// Merge lanes using bitwise OR
const lane = SyncLane | TransitionLanes;
```
Later, you can quickly check if a lane includes a particular priority:
```tsx 
if (lane & SyncLane) {
  // This update includes a sync lane
}
```
</InfoNote>

Since this is a button click and it has the highest priority, so we mark the lane in our `counter` fiber node. Now 
the counter component fiber node holds a lane, now we will propagate all the way up to the root node and merge counter lane 
into the `childLanes` 

<img src="/blog/scheduler/lane.svg" alt="React UI Runtime" width="400" height="100" style={{ display: 'block', margin: '2rem auto' }} />

when we start rendering we start from the root node and begin to process each fiber, `childLanes` helps 
us to tell if the children actually need an update or not. now imagine you are rendering from the top and 
processing each node, you check the `lanes` and `childLanes` and you find both are empty it means we can 
entirely skip that whole subtree 

<InfoNote>
lane just denotes this fiber node needs an update, when react starts the render phase it can check if 
fiber has a lane or not. if it doesn't, no need to re-render this component and move forward
</InfoNote>

Now we marked `lanes` and `childLanes` we will schedule a new render. Remember, we just 
set the priority and scheduled a new render. now when react scheduler sees we have an update 
with a higher priority than rendering a fetch from api it will just halt the rendering of 
that fetch and begin a new re-render 

that's how react fiber architecture does priority based rendering.

in the scheduling phase react also does one important thing, that is it batches the update into a queue which we have discussed in another 
blog 

- [How state updates work internally](https://inside-react.vercel.app/blog/how-state-updates-work-internally)

## Render

React will start with the root fiber and begin to process, and begin to traverse in a child, sibling and parent order
if there is child component go inside the child component if not move to its sibling if there is no sibling and 
no child component move to a parent again, however it will also bailout if there are some conditions


- if a fiber doesn’t have props/context changes and it doesn’t have pending work (empty lanes)
  1. if its children don’t have pending work (empty childLanes), bailout happens and React doesn’t go deeper in the tree.
  2. otherwise React goes straight to its children, without re-rendering this fiber
- otherwise, React tries to re-render it first and then goes to its children.

now react will re-render the `Counter` component and reconcile the children. react will re-run the
function component to get the new children and begin to diff them, after reconciling children of 
`Counter` component react will apply the effect to each child fiber node. 

Once a component re-renders, it generates new React elements, its children all get new props even though they are equal, so React re-renders the whole Fiber Tree by default. Here is why you might need useMemo().

in our case only a text node fiber in the P tag needs an update so in the flag property it will apply 
the update react uses the bitmask operation on the flag property as well.

if you look into the fiber properties you will notice there is a `flags` property, 

```tsx
// this fiber belongs to a textnode inside the p tag

```

## Commit

Commit Phase is the un-interruptable phase of the react where it recursively applies all the changes to 
the dom
